# 选题分析与报告任务化设计

## 目标

将当前首页中的 `内容` Tab 与 `选题分析与报告` Tab 串联起来，形成一套面向内容运营的“多选内容 -> 两阶段 AI 分析 -> 结构化选题洞察”的工作流。

本次设计目标：

- 用户可在 `内容` Tab 中多选想分析的文章或笔记
- 点击 `进行分析` 后创建一条分析任务
- 系统优先抓取更完整的原文，抓不到则退回摘要
- 使用 OpenAI 兼容模型进行两阶段分析
- 在 `选题分析与报告` Tab 中展示任务状态、文章级结构化分析与最终选题洞察
- 每次分析任务保存到 SQLite，挂在当前关键词历史记录下面

## 已确认需求

### 1. 交互需求

- 在关键词搜索后，用户可以在 `内容` Tab 中多选内容
- 页面下方出现 `进行分析` 按钮
- 点击后更新 `选题分析与报告` Tab

### 2. 模型配置

- AI 模型不在前端页面暴露
- 默认 API key 与模型配置放在后端 config 或环境变量中
- 用户无法在页面上手动切换模型

### 3. 分析内容来源

- 优先抓取完整原文
- 如果抓取失败，再退回到当前已保存的摘要与结构化字段

### 4. 保存粒度

- 每次点击 `进行分析` 生成一条独立分析任务
- 分析任务挂在当前这次关键词历史记录下
- 同一个关键词历史记录下可以存在多条分析任务

## 推荐方案

采用“任务化两阶段分析方案”。

### 为什么推荐

- 能完整记录“这份报告是由哪些内容生成的”
- 适合你需要的两阶段分析流程
- 更便于后续追溯、复盘和继续深挖同一话题
- 和当前关键词历史记录、SQLite 结构天然兼容

## 页面交互设计

## 1. 内容 Tab

### 新增能力

- 每篇内容卡片前增加复选框
- 支持多选
- 页面底部出现固定操作栏

### 底部操作栏内容

- 已选内容数
- `清空选择`
- `进行分析`

### 点击 `进行分析` 后的行为

1. 创建一条分析任务
2. 开始两阶段 AI 分析
3. 自动切换到 `选题分析与报告` Tab
4. 展示任务状态为 `分析中`

## 2. 选题分析与报告 Tab

从当前的“占位态”升级为“任务列表 + 任务详情”结构。

### 上部：分析任务列表

每条任务显示：

- 关键词
- 来源平台
- 创建时间
- 状态
- 选中文章数

### 下部：当前任务详情

分成三块：

#### A. 已选内容摘要

逐篇展示第一阶段分析结果：

- 原始标题
- 摘录摘要
- 关键词
- 关键信息
- 亮点
- 可借鉴表达

#### B. 综合洞察

展示第二阶段聚合分析结果：

- 热点总结
- 共性模式
- 差异化机会
- 值得继续跟踪的方向

#### C. 选题建议

至少输出 5 条结构化选题洞察。

每条建议建议固定字段：

- 方向标题
- 选题简介
- 为什么值得做
- 可切入亮点
- 建议参考的原始内容

## 两阶段 AI 分析流程

## 第一阶段：文章级分析

### 输入

- 用户选中的文章/笔记
- 优先使用原文全文
- 抓不到全文则退回：
  - 现有摘要
  - 标题
  - 标签
  - 平台
  - 作者
  - 热度信息

### 输出

每篇内容生成结构化结果：

- 文章摘要
- 关键词
- 原文关键信息
- 内容亮点
- 可借鉴表达方式
- 风险 / 同质化点

## 第二阶段：任务级洞察

### 输入

- 第一阶段所有结构化结果

### 输出

- 热点总结
- 共性模式
- 差异化机会点
- 至少 5 条结构化选题建议

## 后端配置设计

AI 相关配置放在后端，不暴露给前端页面。

建议使用环境变量：

- `AI_BASE_URL`
- `AI_API_KEY`
- `AI_MODEL`

前端不提供模型切换，仅提供 `进行分析` 操作。

## SQLite 设计

基于当前的关键词历史记录体系，新增分析任务相关表。

## 1. analysis_tasks

表示一次用户点击 `进行分析` 产生的任务。

字段建议：

- `id`
- `search_id`
- `source_type`
- `status`
  - `pending`
  - `running`
  - `completed`
  - `failed`
- `selected_count`
- `created_at`
- `updated_at`

## 2. analysis_task_items

表示该任务下选中了哪些内容。

字段建议：

- `id`
- `task_id`
- `content_source_type`
- `content_item_id`
- `content_title`
- `original_url`
- `full_text_status`
  - `success`
  - `fallback_summary`
  - `failed`
- `created_at`

## 3. analysis_item_summaries

保存第一阶段每篇内容的结构化分析结果。

字段建议：

- `id`
- `task_item_id`
- `summary`
- `keywords_json`
- `key_points_json`
- `highlights_json`
- `angles_json`
- `risks_json`
- `created_at`

## 4. topic_reports

从“关键词历史占位”升级为真正与分析任务关联。

建议调整：

- 增加 `task_id`
- 一条分析任务对应一份报告

字段中保存：

- `report_status`
- `summary`
- `hot_insights`

## 5. topic_suggestions

继续保存最终选题建议。

每条建议建议增加字段：

- `title`
- `brief`
- `why_now`
- `entry_point`
- `reference_items_json`
- `created_at`

## 关系设计

- 一次关键词历史记录可以有多次分析任务
- 一次分析任务可以选择多篇内容
- 每篇内容有一份文章级结构化分析
- 一次分析任务最终产生一份报告和多条选题建议

## 原文抓取策略

优先级如下：

1. 有原文链接时尝试抓原文
2. 抓取成功则使用原文全文
3. 抓取失败则退回当前已保存摘要
4. 若摘要过短，则组合标题、标签、平台、互动数据作为补充上下文

## 风险与注意点

### 1. 原文抓取稳定性

原文抓取可能受平台限制，因此必须支持摘要回退。

### 2. 分析耗时

两阶段分析比单次调用更耗时，需要清晰展示任务状态。

### 3. 结果可追溯性

必须把“选中了哪些内容”与“生成了哪份报告”绑定保存，避免报告不可复盘。

## 验收标准

- 用户能在 `内容` Tab 多选内容
- 点击 `进行分析` 后创建一条分析任务
- 任务挂在当前关键词历史记录下
- `选题分析与报告` Tab 能显示任务状态
- 分析完成后能看到文章级结构化分析
- 最终至少输出 5 条结构化选题洞察
